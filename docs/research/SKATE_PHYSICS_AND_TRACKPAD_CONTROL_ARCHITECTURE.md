# Skate-like Physics and Trackpad Control Architecture for SlackPad 360

**Status:** Research report and implementation proposal  
**Date:** 2026-07-15  
**Repository audited:** `coldshalamov/SlackPad-360`  
**Audit baseline:** [`f019267917a87f3860c84fc4fe251ccbdd076cfc`](https://github.com/coldshalamov/SlackPad-360/commit/f019267917a87f3860c84fc4fe251ccbdd076cfc)  
**Scope:** Ground locomotion, skateboard/rider dynamics, pop/flip/catch/landing/grind mechanics, trackpad control semantics, numerical robustness, calibration, observability, automated playtesting, and a staged migration path.

> **Verdict:** Keep the deterministic harness, replay system, and Rapier integration for now. Replace the *causal model* around them. SlackPad currently treats the trackpad as a heading-and-trick command surface acting on a very light board-only rigid body. A Skate-like system should treat the trackpad as two feet controlling a rider–board system: foot trajectories and rider center-of-mass motion create forces; those forces load the trucks and wheels; the resulting contacts move the board. The difference is not polish. It is the direction in which causality flows.

---

## 1. Executive conclusion

SlackPad already has several foundations that are better than many prototypes:

- a fixed-step deterministic simulation;
- a single simulation authority;
- high-rate native contact samples;
- replay/checkpoint hashing;
- an agent harness that uses the same input path as hardware;
- explicit, bounded impulses rather than pose teleportation;
- a useful separation between input recognition, commands, physics, and presentation.

Those are worth preserving.

The game feels physically wrong because the current board model omits the dominant mass, compliance, and control source: the rider. It then compensates with direct yaw servos, motorized wheel drive, target angular velocities for tricks, catch damping, rotation quantization, landing speed deletion, and spring-assisted grinds. These systems are internally coherent as an arcade controller, but they do not reproduce the mechanics that make *Skate* feel grounded.

The strongest direct evidence about EA Black Box's original *Skate* is unusually clear. Its team described:

- gesture input sampled at 120 Hz;
- physics simulated at 60 Hz;
- articulated rider joints;
- physically modeled trucks, wheels, and hinges;
- pop generated by forces;
- animation used as a target for an active physical body, blended toward ragdoll rather than replacing dynamics.

That is the architectural north star. The exact proprietary friction, constraint, and controller code is not public. Anyone claiming the exact *Skate* solver can be reproduced from public information is selling mist in a bottle. What is public is sufficient to reproduce the important design principles.

The recommended runtime architecture is a **hybrid articulated rider–skateboard model**:

1. Keep Rapier as broadphase, rigid-body, collision-query, and integration infrastructure.
2. Replace the board-only model with:
   - a deck rigid body;
   - articulated or reduced-order front and rear truck states;
   - four wheels with explicit spin and slip state;
   - a minimal dynamic rider proxy with realistic mass and inertia;
   - two compliant foot contacts driven by trackpad trajectories.
3. Treat gestures as desired foot and center-of-mass trajectories, not trick labels that directly set board angular velocity.
4. Make every assist an explicit, bounded force/impulse/energy contribution, visible in traces.
5. Build a system-identification and playtesting loop before tuning “feel.”

This can be delivered incrementally. A full humanoid ragdoll is not required for the first major realism gain. A low-dimensional rider with pelvis/torso mass, two leg springs, two task-space feet, and phase-aware contacts will fix more of the game than an engine swap or months of tuning the current servos.

It does **not** expose their complete source code, contact solver, friction curves, or controller gains.

## 4. Current SlackPad architecture: what is good

Before discussing failures, it is important to preserve what already works.

### 4.1 Deterministic simulation authority

`SimWorld` has one fixed simulation clock, one integrator call, deterministic construction order, and seeded randomness. That is excellent infrastructure for physics development. Realism work without deterministic reproduction quickly becomes séance engineering: every regression has a new ghost.

### 4.2 Hardware and agents share the same input path

`AgentHarness` deliberately denies agents direct pose writes or hidden force APIs. Agents inject `ContactFrame`s and observe copied state. This is exactly the right anti-cheating contract for automated playtesting.

### 4.3 High-rate input is already retained

The control trace records the calibrated samples consumed by each 60 Hz simulation step, including the approximately 125 Hz foot samples. This is close to the original *Skate* principle of preserving gesture profiles at a higher rate than the physics tick.

### 4.4 Existing code already avoids many crude cheats

Pop, flip, catch, landing, and grind assistance are generally expressed as impulses or torques rather than direct pose teleports. This makes the system repairable: the problem is mostly *what forces are generated and why*, not a complete absence of dynamics.

### 4.5 The config is honest

The configuration file explicitly calls many values hypotheses and tuning candidates. That is healthier than false precision. The next step is to replace hypothesis-only tuning with measured system identification and trace-based acceptance tests.

## 5. Why the game feels wrong

The realism gap is not one bug. It is a stack of compensations for an incomplete physical model.

### 5.1 Steering has two competing causal paths

There is a physically motivated path:

```text
board lean -> truck steering -> wheel contact forces -> yaw
```

But there is also a much stronger direct path:

```text
two-finger absolute angle -> desired world heading -> yaw-rate servo -> body torque -> yaw
```

`BoardController` converts the absolute tail-to-nose finger segment into an absolute world-space board heading. `SimWorld.applyGroundForces()` then computes heading error and applies a direct yaw torque, while the wheel solver separately steers the trucks from deck lean.

This creates several nonphysical behaviors:

- the board can yaw at or near zero forward speed;
- input can rotate the deck without first producing lean or lateral contact force;
- truck geometry becomes secondary to a chassis orientation controller;
- the player is effectively aiming a rigid body, not loading a skateboard;
- the wheel solver must then annihilate lateral velocity to make the aimed board travel in the new direction.

A skateboard is a nonholonomic system: at ordinary riding speeds, yaw is a consequence of forward motion, lean, truck geometry, and contact forces. The direct heading servo violates that structure.

#### Quantitative check

The configured board is approximately a cuboid with:

- mass \(m = 2.4\,\mathrm{kg}\);
- length \(L = 0.8\,\mathrm{m}\);
- width \(W = 0.2\,\mathrm{m}\).

Its approximate yaw inertia is

\[
I_y \approx \frac{m(L^2 + W^2)}{12}
= 0.136\,\mathrm{kg\,m^2}.
\]

The configured maximum direct yaw torque is \(28\,\mathrm{N\,m}\). Therefore,

\[
\alpha_y = \frac{\tau}{I_y}
\approx 206\,\mathrm{rad/s^2}.
\]

At 60 Hz, a single step can change yaw rate by roughly

\[
\Delta\omega_y \approx 3.43\,\mathrm{rad/s}.
\]

The configured target yaw rate can reach \(10\,\mathrm{rad/s}\).

By contrast, a simple skateboard/bicycle approximation gives

\[
\dot{\psi} \approx \frac{v}{L_w}\tan\sigma,
\]

where \(L_w = 0.43\,\mathrm{m}\) is wheelbase and \(\sigma\) is truck steer. At the configured maximum \(\sigma=18^\circ\):

| Forward speed | Kinematic yaw rate |
|---:|---:|
| 0 m/s | 0 rad/s |
| 0.5 m/s | 0.38 rad/s |
| 1.0 m/s | 0.76 rad/s |
| 2.0 m/s | 1.51 rad/s |
| 5.5 m/s | 4.16 rad/s |
| 8.0 m/s | 6.05 rad/s |

The servo can therefore dominate the physically plausible carve rate, especially at low speed. This is the clearest single explanation for “the board does not feel connected to the ground.”

### 5.2 The dominant mass—the rider—is absent

The board is one 2.4 kg dynamic body. The virtual shoes are presentation/observation state, not masses connected through forces.

That omission affects almost every regime:

- cruise acceleration;
- push acceleration;
- truck compression;
- carve response;
- load transfer;
- pumping;
- pop preparation;
- body rotation;
- catch;
- landing;
- bail thresholds.

A 70 kg rider plus a 2.4 kg board weighs about 710 N. The current suspension model is scaled by board mass and capped at 80 N per wheel, or 320 N total. It cannot statically represent a rider-loaded skateboard even if a rider mass were visually implied.

This is not fixed by changing `boardMass` from 2.4 kg to 72 kg. During a real trick, the rider and board may be strongly coupled before pop, partially coupled during flick, separated in flight, then coupled again at catch and landing. The effective inertia changes with contact topology. A single “rider-added board mass” would improve cruising and destroy flips.

The correct fix is a separate rider body or reduced rider model connected by explicit contacts.

### 5.3 Propulsion is a motor, not a push

In the current shipping contract, Ctrl authorizes continuous cruise drive. `SimWorld` sends the drive scalar into the wheel solver, which applies it independently at every contacted wheel. Unless the config value is intentionally per-wheel, the effective total drive can be approximately four times the named scalar. This semantic should be audited.

At low speed, a configured 20 N applied at four wheels can create approximately 80 N total drive on a 2.4 kg body—about \(33\,\mathrm{m/s^2}\) before other limits. The high friction cap permits much of that force.

A real push is a hybrid contact event:

1. one foot remains on the board;
2. the rider shifts weight;
3. the pushing foot contacts the ground;
4. tangential ground reaction accelerates the rider–board system;
5. the pushing foot lifts and returns;
6. the system glides with rolling resistance.

The current system skips the rider and the ground foot. It accelerates a powered board. This is a valid accessibility mode, but it should not be the raw physics mode.

The one-shot configured `pushImpulse = 2.9 N·s` gives a 2.4 kg board an ideal \(\Delta v \approx 1.21\,\mathrm{m/s}\). Applied to a 72.4 kg rider–board system, the same impulse would produce only \(\approx 0.04\,\mathrm{m/s}\). The current impulse scale is therefore compensating for missing rider mass.

### 5.4 The wheel model has no wheel dynamics

`SkateboardContactSolver` uses four ray-cast contacts. For each wheel it computes:

- a suspension spring/damper support impulse;
- a rolling direction;
- longitudinal velocity;
- lateral velocity;
- a capped longitudinal impulse;
- a capped lateral cancellation impulse;
- a visual wheel angle integrated from ground speed.

What it does **not** model:

- wheel angular inertia;
- torque balance;
- longitudinal slip ratio;
- lateral slip angle;
- a coupled friction ellipse;
- finite wheel shape at curbs and edges;
- bearing drag as a function of wheel speed/load;
- wheel deformation;
- truck bushing preload/hysteresis;
- load-sensitive grip.

The `rotation` value is kinematic presentation state. It does not feed back into traction.

This matters because a skateboard's feel is dominated by the transition between rolling, carving, scrubbing, sliding, and impact. A single lateral-velocity cancellation law can be stable, but it tends to feel binary.

With `wheelSideFrictionStiffness = 30` and 60 Hz:

\[
\text{lateralGain}=\mathrm{clamp}(30/60,0,1)=0.5.
\]

If all four wheels are loaded and the friction cap does not bind, the solver removes about 50% of lateral chassis velocity in one frame. The equivalent exponential time constant is about 24 ms. That is nearly an instantaneous rail constraint. Combined with the direct yaw servo, the board can “aim, then snap” rather than carve.

### 5.5 The trucks are a steering function, not a mechanical subsystem

The current truck steering function maps deck lean to front/rear wheel steer. That is directionally correct, but the truck has no independent state or identified compliance. Real trucks include:

- kingpin/pivot geometry;
- elastic bushings;
- preload;
- nonlinear stiffness;
- velocity-dependent damping;
- Coulomb-like pivot friction;
- mechanical stops;
- front/rear asymmetry;
- load dependence.

The HUSKY work demonstrates a useful minimum viable abstraction:

\[
\tan \sigma = \tan \lambda \sin \gamma,
\]

where \(\gamma\) is board tilt, \(\lambda\) is truck rake geometry, and \(\sigma\) is induced steering. It also identifies equivalent truck stiffness and damping from a real board's free-decay roll response.

SlackPad currently has neither an articulated truck state nor measured parameters. It has a geometry-inspired output plus a separate yaw servo.

### 5.6 Tricks are controlled angular-velocity programs

The trick pipeline is physically cleaner than a pose animation, but it is still strongly command-driven.

For flips:

1. a gesture is classified;
2. the classifier produces a target angular speed;
3. a one-shot angular impulse moves the board toward that target;
4. a per-step PD torque continues tracking it;
5. catch removes angular velocity;
6. quantization removes more on-axis rotation near a whole trick.

For shuvs, a similar program acts around board up.

The most revealing line is in `AirGestureClassifier`: once the forgiving flip gate is crossed, intensity is forced into the range

\[
s = 0.72 + 0.28\,s_{\text{raw}}.
\]

A barely accepted flick therefore asks for at least 72% of maximum flip angular speed. For shuvs, the target is effectively the configured maximum with direction.

The board's approximate long-axis inertia is

\[
I_{\text{long}} \approx
\frac{m(W^2+t^2)}{12}
= 0.0085\,\mathrm{kg\,m^2}.
\]

The configured flip impulse caps of 0.08, 0.14, and 0.20 N·m·s correspond to ideal angular-speed changes of approximately:

| Assist level | Impulse cap | Ideal \(\Delta\omega\) |
|---:|---:|---:|
| 0 | 0.08 N·m·s | 9.4 rad/s |
| 1 | 0.14 N·m·s | 16.5 rad/s |
| 2 | 0.20 N·m·s | 23.5 rad/s |

The maximum flip target is 15 rad/s. At standard assist, the one-shot budget can move the lightweight deck to the full target in essentially one physics step, after which PD guidance maintains the program. The player's path mostly decides *which program opens*, not the complete momentum transfer.

That is why tricks can feel recognized rather than performed.

### 5.7 Pop is board-only and cannot model unweighting

The current pop is better than a teleport: it applies upward impulse plus a downward point impulse at the selected end, creating pitch through a lever arm. At 2.4 kg, the configured 5.8–9.6 N·s vertical impulse produces ideal board-only launch speeds of 2.42–4.0 m/s and ballistic heights of roughly 0.30–0.82 m.

The missing mechanics are:

- crouch and rider center-of-mass lowering;
- rapid unweighting;
- tail-ground collision;
- front-foot slide;
- board/rider separation;
- transfer of angular momentum between rider and board.

A real ollie is not “add vertical impulse to board.” It is a timed contact sequence. The current result can have plausible height while still feeling wrong because the cause, mass distribution, and timing are wrong.

### 5.8 Catch, landing, and grind assistance silently remove or add momentum

Examples:

- catch applies angular damping;
- catch quantization removes on-axis spin inside cones up to 80 degrees at high assist;
- dirty landing deletes 35% of horizontal speed;
- grind latch applies lateral springs, yaw alignment, roll alignment, tangent drag, and dismount impulses.

Assistance is not inherently bad. *Skate* itself used active physical control rather than demanding raw passive dynamics. The problem is hidden authority. If the system adds or removes substantial momentum without recording the work and without a clear player-facing profile, the player cannot build a stable internal model.

The solution is not “zero assists everywhere.” It is:

- raw mode with near-zero hidden work;
- standard mode with bounded, traceable assistance;
- accessible mode with larger but still explicit envelopes;
- an energy/impulse ledger that exposes every assist contribution.

### 5.9 Recognition is more observable than the contact mechanics

The trace already records raw contacts, calibrated feet, intent, board pose/velocity, phase, camera, and outcome. That is a strong start.

It does not record the variables needed to explain physical feel:

- per-wheel normal load;
- suspension compression and velocity;
- wheel angular speed;
- slip ratio and slip angle;
- longitudinal/lateral contact impulses;
- truck tilt, steer, bushing torque, and stop contact;
- rider center of mass;
- foot contact forces and modes;
- total kinetic/potential energy;
- work done by each controller and assist;
- solver residuals and penetration;
- phase-transition causes.

The present observability can explain *what label opened*. It cannot fully explain *why the board moved that way*.

## 6. What the commercial games actually tell us

### 6.1 EA Black Box's *Skate* (Xbox 360/PS3)

#### Confirmed

A contemporary interview with executive producer Scott Blackwood describes the following:

- Flick-It began as a gesture-recognition prototype.
- Gesture speed and accuracy were meaningful.
- Stick input was sampled at 120 Hz because 60 Hz missed fast profiles.
- Physics ran at 60 Hz.
- The physics system was built on RenderWare “Drives.”
- Drives modeled rider joints and skateboard components including trucks, wheels, and hinges.
- Pop was produced by physical forces.
- Motion-capture animation served as a target for a physically simulated body, blended between active drive and ragdoll behavior.
- The team evaluated middleware and built custom “special sauce” where required.

#### What this implies

*Skate* did not get its feel from one magic tire coefficient. It combined:

1. high-rate gesture capture;
2. a physically articulated rider and board;
3. active physical control;
4. animation as a target, not final authority;
5. a solver/controller tuned around the complete coupled system.

This is directly transferable to SlackPad:

- 125 Hz trackpad samples are adequate to preserve gesture shape;
- the missing piece is turning those samples into foot/rider targets and forces;
- the renderer should follow physics;
- the active controller should stabilize but not overwrite contact causality.

#### What remains unknown

Public sources do not reveal:

- exact constraint solver implementation;
- exact wheel friction law;
- exact truck bushing model;
- exact rider joint gains;
- exact assistance envelopes;
- complete source code.

The goal should therefore be **behavioral and architectural reproduction**, not unsupported claims of source-level cloning.

### 6.2 *Skater XL*

The official description says:

- each thumbstick maps to the corresponding foot;
- there is no pre-programmed trick list in the control system;
- board response is continuously driven by stick movement.

This is official design intent, not proof of every internal force law. The important lesson is control bandwidth: a player manipulates two effectors continuously rather than selecting a trick macro.

For SlackPad, two touch contacts are an unusually natural analogue:

```text
left/right or tail/nose finger
    -> desired foot placement and velocity
    -> compliant foot-board interaction
    -> board response
```

The mistake would be to copy only the phrase “independent feet” while continuing to convert those feet into direct board orientation and target spin.

### 6.3 *Session: Skate Sim*

Public information supports that *Session* uses a physics-based, dual-stick simulation philosophy and Unreal Engine 4. The exact contact and control implementation is proprietary.

A revealing engineering fact is the developer's reported cancellation of an Unreal Engine 5 migration after concluding that integrating the existing physics and core gameplay into the changed physics engine would be too costly and might not preserve feel. The lesson is not “UE4 is the best engine.” It is that mature game feel becomes tightly coupled to contact details, timestep behavior, controller gains, and numerical quirks.

Therefore SlackPad should not swap Rapier first and hope realism appears. A physics-engine migration before defining physical scenarios, observability, and acceptance metrics would exchange known errors for unknown errors.

### 6.4 *Tony Hawk's Pro Skater* and *Underground*

The original *Tony Hawk* lineage is a different design species.

Developer history describes:

- automatic forward propulsion inspired by racing games;
- button-authored trick vocabulary;
- extreme emphasis on responsiveness;
- camera behavior tightly coupled to controller feel;
- levels and physics consciously shaped for flow and fun rather than physical fidelity.

No authoritative open-source release of the original Neversoft movement/physics code was found. Community projects such as THUG Pro preserve or modify game behavior, but they are not a clean public specification of the original solver.

The correct lesson to borrow is not the dynamics. It is presentation and flow:

- input should respond immediately;
- the camera should reveal takeoff, trajectory, and landing;
- failure should be legible;
- level geometry should advertise lines;
- recovery should preserve rhythm;
- controls should form a learnable movement language.

SlackPad should use *Skate* as the physical reference and *Tony Hawk* as the legibility/reference-flow reference.

#### Comparison matrix

| Game family | Primary control idea | Physical architecture publicly supported | Best lesson for SlackPad |
|---|---|---|---|
| *Skate* | gesture-driven body/board control | articulated rider and board; active-drive physics; force pop | causal architecture and tactile weight |
| *Skater XL* | continuous independent feet | exact internals proprietary | continuous foot authority |
| *Session* | deliberate dual-foot simulation | exact internals proprietary; physics/core tightly engine-coupled | consistency, difficulty, engine/tuning discipline |
| *Tony Hawk/THUG* | arcade movement vocabulary | self-propelled, state/animation-driven design | response, camera, flow, readable failure |

## 7. What the research literature and open source add

### 7.1 HUSKY: the strongest open implementation

The 2026 HUSKY paper and MIT-licensed repository provide the most directly relevant public reference found.

Its skateboard model includes:

- an articulated deck tilt joint;
- front and rear truck roll joints;
- four wheel joints;
- explicit deck/truck/wheel masses;
- coupling between deck tilt and truck motion;
- anisotropic wheel friction;
- explicit foot-ground, foot-board, and wheel-ground contact sensing;
- a 23-DoF humanoid controller;
- phase-specific push, steering, and transition objectives;
- a 5 ms physics timestep;
- domain randomization of center of mass and friction.

Its paper formalizes skateboarding as:

- nonholonomic;
- underactuated;
- tightly coupled between rider and board;
- hybrid, because contact topology changes between pushing and steering.

Most importantly, it demonstrates practical physical identification of truck stiffness and damping from free-decay roll.

SlackPad does not need to copy the full humanoid RL stack. It should copy the decomposition:

- articulated or reduced skateboard mechanics;
- explicit rider-board contacts;
- phase-aware control;
- measured truck parameters;
- uncertainty-aware testing.

### 7.2 Aalto MuJoCo MPC skateboarding thesis

Elias Mikkola's thesis builds a simplified humanoid plus mechanically approximated skateboard in MuJoCo MPC. It demonstrates pushing and steering through reference-pose control and contact mechanics, while reporting limited success on ollies.

That negative result is valuable. Locomotion can be solved with a reduced model and carefully designed objectives. A convincing ollie is substantially harder because it demands accurate hybrid contact timing, force transfer, and separation. SlackPad should stage development in the same order instead of trying to perfect every trick while the ride model is still wrong.

### 7.3 Mechanical optimization of pumping

Kogelbauer et al. model half-pipe pumping as deliberate rider center-of-mass modulation and compare the optimal timing with skilled and unskilled riders. Skilled behavior follows the predicted timing more closely.

This supports a central design requirement: rider COM state is not cosmetic. It is a control variable that creates speed and stability. A board-only model cannot reproduce pumping or realistic transition behavior regardless of camera quality.

### 7.4 Differentiable and high-throughput simulators

Nimble, Dojo, MuJoCo/MJX, and Isaac Gym/Lab are useful as **offline laboratories**:

- parameter identification;
- sensitivity analysis;
- trajectory optimization;
- controller training;
- domain randomization;
- large scenario sweeps.

They should not automatically replace the browser runtime. The robust architecture is:

```text
high-fidelity/offline model
    -> fit parameters and policies
    -> export compact deterministic controller/model
    -> run in SlackPad/Rapier
    -> verify against the same scenarios
```

This separates research throughput from shipping constraints.

## 8. Definition of a maximally robust system

“Robust” should not mean “most complicated.” It should mean the system remains understandable and stable across:

- different trackpads and sample jitter;
- different frame rates;
- different physics substeps;
- regular and goofy stance;
- different rider masses;
- different board/truck/wheel setups;
- flat ground, banks, transitions, curbs, rails, and impact;
- imperfect gestures;
- assist levels;
- replay and headless execution;
- automated parameter search.

The target system should satisfy these principles.

### 8.1 Physical causality

Every change in linear or angular momentum must be attributable to:

- gravity;
- contact;
- an actuator/controller force;
- a documented game assist.

No invisible heading torque, angular-velocity assignment, or speed deletion should exist in raw mode.

### 8.2 Equal and opposite reaction

A foot force on the board must apply the opposite force to the rider. A ground push must accelerate the rider-board system through ground reaction. A catch must transfer momentum between board and rider.

### 8.3 Hybrid contact modes

The controller should explicitly model:

- glide/carve;
- push;
- crouch/pop preparation;
- tail strike;
- board separation;
- flick/drag contact;
- flight;
- catch;
- landing;
- grind/slide;
- bail.

The modes may be estimated probabilistically, but the contact topology should not be implicit in a generic “ground/air” flag.

### 8.4 Identifiable parameters

Important parameters must correspond to measurable quantities:

- rider mass and inertia;
- board mass/inertia;
- truck stiffness and damping;
- truck geometry;
- wheel radius/inertia;
- rolling resistance;
- wheel friction;
- leg stiffness/damping;
- foot friction/compliance.

A value should have a test that can estimate or falsify it.

### 8.5 Bounded active control

Active-drive controllers are allowed and necessary. Their force, torque, impulse, power, and work must be capped and logged.

### 8.6 Numerical convergence

A maneuver should not qualitatively change because the contact solver runs at 60, 120, or 240 Hz. Results need not be bit-identical across configurations, but metrics should converge within defined tolerances.

### 8.7 Graceful degradation

When contact confidence is low or samples drop:

- hold a short predicted trajectory;
- decay toward neutral;
- do not inject a large impulse;
- expose degraded confidence in telemetry;
- recover without role swapping.

## 9. Recommended runtime physics architecture

### 9.1 Keep Rapier, replace the model around it

The first implementation should retain Rapier for:

- deterministic fixed-step integration;
- static environment collision;
- deck/rider rigid bodies;
- broadphase and shape queries;
- contact events;
- replay compatibility.

Do not begin with an engine migration. The current dominant errors come from control and modeling, not evidence that Rapier cannot integrate the required rigid bodies.

Use a feature-gated architecture so the current model remains a baseline:

```text
PhysicsModel = "legacy-board" | "rider-board-v1" | "rider-board-v2"
```

### 9.2 State decomposition

A practical first target:

#### Deck

- one rigid body;
- realistic board mass and inertia;
- deck and tail/nose collision geometry;
- no artificial rider mass baked into it.

#### Front and rear trucks

Either:

1. articulated truck bodies/joints in Rapier; or
2. reduced-order scalar states integrated by a custom force model.

For a browser game, reduced-order truck states are likely the best first step:

```ts
interface TruckState {
  roll: number;
  rollRate: number;
  steer: number;
  normalLoad: number;
  bushingTorque: number;
  atStop: boolean;
}
```

#### Wheels

Each wheel should carry:

```ts
interface WheelState {
  omega: number;
  angle: number;
  normalLoad: number;
  compression: number;
  compressionRate: number;
  slipRatio: number;
  slipAngle: number;
  longitudinalImpulse: number;
  lateralImpulse: number;
  contactPoint: Vec3 | null;
  contactNormal: Vec3 | null;
}
```

The wheel need not be a separate Rapier rigid body initially. Explicit spin state plus swept-sphere contact is enough.

#### Minimal rider

A useful `RiderProxyV1`:

- pelvis/torso rigid body, or one composite body with 6-DoF pose;
- configurable mass, center of mass, and inertia;
- two foot effectors;
- two compliant leg constraints;
- optional torso yaw/roll targets;
- no detailed knees/elbows required in v1.

The rider proxy should be invisible or rendered procedurally at first. Its job is dynamics.

### 9.3 Truck model

Use geometry plus compliance.

A lean-to-steer relation:

\[
\sigma_i =
s_i\,\arctan\left(\tan\lambda_i \sin\gamma_i\right),
\]

where \(s_i\) gives the front/rear sign convention.

Model bushing torque with a stable, identifiable law:

\[
\tau_b =
-k_1\phi
-k_3\phi^3
-c\dot{\phi}
-\tau_c\tanh(\dot{\phi}/\epsilon)
-\tau_{\text{stop}}(\phi).
\]

Start with \(k_3=\tau_c=0\) if data is sparse. Add nonlinear terms only when measured decay/carve tests require them.

`tau_stop` should be a soft unilateral stop near maximum lean, not a hard state clamp.

Front/rear trucks should permit different:

- geometry;
- stiffness;
- damping;
- preload;
- stop angle.

#### System identification

Measure free-decay roll:

1. place real board on a fixture or controlled load;
2. perturb roll;
3. record angle with phone IMU/video;
4. extract successive peaks and period;
5. estimate damping ratio from logarithmic decrement;
6. estimate torsional stiffness from natural frequency;
7. fit friction/nonlinearity from residuals.

This is inexpensive and far more informative than hand-tuning a “carve gain.”

### 9.4 Wheel and ground contact

#### Contact geometry

Use swept sphere/capsule/cylinder queries rather than only a center ray. A ray is stable on flat ground but does not represent:

- curb strikes;
- wheel climbing;
- edge contact;
- partial wheel support;
- sidewall/axle interaction.

A robust compromise is:

- shape cast wheel sphere along suspension direction;
- contact manifold approximation at hit;
- explicit deck/truck collider contact for non-wheel impacts.

#### Normal force

A compliant normal model:

\[
F_z = \max\left(0,\,
k_s x + c_s \dot{x}
\right),
\]

with compression/relaxation damping and a smooth force cap. Use load-aware parameters and substeps.

#### Wheel spin

\[
I_w\dot{\omega}
=
\tau_{\text{bearing}}
-
R F_x.
\]

The board is not motor-driven in raw mode. Longitudinal force emerges from slip and chassis motion.

#### Slip

Longitudinal slip ratio:

\[
\kappa =
\frac{R\omega - v_x}
{\max(|v_x|,v_\epsilon)}.
\]

Lateral slip angle:

\[
\alpha =
\operatorname{atan2}(v_y, |v_x|+v_\epsilon).
\]

A regularized brush-like force law is sufficient:

\[
F_x^0 = C_\kappa \kappa,\qquad
F_y^0 = -C_\alpha \tan\alpha.
\]

Project the requested force into an anisotropic friction ellipse:

\[
\left(\frac{F_x}{\mu_x F_z}\right)^2
+
\left(\frac{F_y}{\mu_y F_z}\right)^2
\le 1.
\]

This gives continuous transitions between rolling, carving, scrubbing, and sliding. Pacejka-style complexity is not justified without tire data.

#### Rolling resistance

Use bearing/urethane loss rather than global chassis drag alone:

\[
\tau_{\text{roll}}
=
-\left(\tau_0 + c_\omega|\omega|\right)
\tanh(\omega/\epsilon).
\]

Surface material can scale \(\mu_x,\mu_y\), restitution, and rolling loss.

### 9.5 Rider and foot contact controller

The trackpad should command desired task-space trajectories.

For foot \(i\):

\[
F_i^* =
K_p(x_i^* - x_i)
+
K_d(v_i^* - v_i),
\]

then project/clamp according to contact mode, friction, leg reach, force, power, and work budgets.

Apply:

- \(+F_i\) to the contacted body;
- \(-F_i\) to the rider;
- torque from the contact lever arm to both.

A foot contact should include:

- normal compliance;
- tangential friction;
- torsional friction;
- release condition;
- recontact hysteresis;
- confidence.

The minimal rider controller should also have a desired pelvis/COM trajectory derived from trackpad centroid and phase. It should be an active physical controller, not a board orientation controller.

### 9.6 Ground locomotion modes

#### Glide/carve

- both feet on board;
- no wheel motor;
- COM lateral shift creates deck roll;
- truck geometry creates steering;
- wheel forces create yaw;
- fore/aft weight shift changes truck load and stability.

#### Push

- one foot remains on board;
- the other follows a backward ground-sweep trajectory;
- ground contact produces tangential impulse;
- board and rider accelerate together;
- push magnitude comes from sweep speed, duration, virtual leg force, and available friction;
- the foot returns to the board through a transition trajectory.

#### Pump

- COM movement normal to a transition exchanges work with the ramp;
- no hidden forward engine is required;
- pumping can be tested against measured half-pipe energy gain.

#### Low-speed pivot/tic-tac

At near-zero speed, ordinary truck steering cannot yaw the board significantly. Low-speed turning should therefore be a separate physical mode:

- unload/lift front or rear wheels;
- apply foot torque through a pivot contact;
- rotate around the loaded truck/tail;
- require an explicit weight-transfer gesture.

This replaces the current standstill heading servo with a real skateboard behavior.

### 9.7 Trick mechanics

#### Ollie/nollie

A physically causal sequence:

1. crouch lowers rider COM and stores controller/leg work;
2. rider unweights;
3. rear/front foot drives the tail/nose;
4. tail/nose collides with ground;
5. board gains pitch and vertical velocity;
6. leading foot slides along deck;
7. foot force levels the board;
8. feet separate or maintain light compliant contact;
9. rider follows a controlled ballistic trajectory.

Pop quality should emerge from:

- crouch depth;
- unweight timing;
- tail strike speed;
- foot placement;
- front-foot slide timing;
- available leg force.

A trick label may explain the event; it must not create the event.

#### Kickflip/heelflip

- front-foot target moves diagonally toward/over an edge;
- while foot-board contact exists, tangential force and contact lever arm create roll impulse;
- release occurs when contact geometry separates;
- after release, board angular momentum evolves passively except for air drag/negligible effects;
- raw mode has no midair target-angular-velocity PD.

The gesture's entire time history matters: path, speed, edge crossing, contact duration, and release timing.

#### Shuv

- rear foot applies a tangential sweep at the tail;
- contact lever arm creates yaw impulse;
- rider receives equal/opposite reaction;
- board rotation depends on actual impulse and yaw inertia.

#### Body rotation

A rider body turn should arise from:

- pre-takeoff torso/hip torque;
- foot-board reaction;
- angular momentum at separation.

The board and rider should not receive independent magical spin commands.

#### Catch

- a foot intersects/contact-tests the deck;
- a compliant catch constraint transfers momentum;
- the board may be caught one-foot first;
- excessive relative speed or poor placement causes bounce/miss/bail;
- standard assist may enlarge foot capture radius or add bounded damping;
- all assist impulse is logged.

#### Landing

- wheel/deck contacts generate impulses;
- rider legs compress;
- rider COM and board orientation determine stability;
- residual board/rider relative motion matters;
- bails arise from support polygon, impulse, angle, and controller saturation.

Do not delete a fixed fraction of horizontal speed as the primary dirty-landing mechanic. Let tire scrub, deck strike, leg work, and instability dissipate energy.

### 9.8 Grinds and slides

Use actual contact manifolds and material-dependent anisotropic friction.

For a 50-50:

- truck hanger/axle geometry contacts rail;
- rail tangent has low friction;
- lateral separation/roll is constrained only by real contact plus optional assist.

For a boardslide:

- deck contacts rail;
- along-rail friction is low;
- across-rail friction and yaw behavior follow the contact patch.

Assisted profiles may use a capture envelope, but:

- candidate state must be visible before latch;
- assist force/torque must be bounded;
- no pose snap;
- work must be recorded;
- raw mode should allow zero capture force.

The current grind candidate/latch separation is a good design to preserve.

## 10. Trackpad controls: feet and body, not virtual joysticks

A trackpad can be better than two sticks for this problem because it provides:

- absolute 2D contact positions;
- multiple contacts;
- high-rate path samples;
- lift/replant timing;
- contact identity;
- path curvature;
- motion speed and acceleration.

It currently does **not** reliably provide pressure in the native raw path. The shared schema permits pressure/width/height, but the audited HID adapter emits position, identity, tip, confidence, and timing. The design must not invent pressure.

### 10.1 Calibrated coordinate frame

At setup:

1. player places two fingers in neutral stance;
2. infer tail/nose roles;
3. estimate pad-to-board rotation;
4. record neutral separation and centroid;
5. estimate comfortable movement bounds;
6. store regular/goofy mirroring exactly once.

Use a continuous calibrated frame:

```ts
interface FootTargetSample {
  t: number;
  role: "front" | "rear";
  position: Vec2;       // normalized player-calibrated space
  velocity: Vec2;
  acceleration: Vec2;
  plantedIntent: boolean;
  confidence: number;
}
```

Do not repeatedly reinterpret stance in multiple layers.

### 10.2 Separate common-mode and differential motion

For contacts \(p_f,p_r\):

\[
c = \frac{p_f+p_r}{2}
\]

is common-mode centroid.

\[
d = p_f-p_r
\]

is relative stance vector.

Use these as *intent features*, not direct board state:

- lateral centroid shift -> desired rider COM lean;
- fore/aft centroid shift -> desired pelvis/load distribution;
- stance length -> crouch/extension cue and foot spacing;
- stance angle change -> torso/hip yaw intent;
- individual foot motion -> foot target trajectory;
- lift/replant -> contact-mode transition.

The current mapping makes the stance angle authoritative world heading. The proposed mapping makes it one bounded cue to body orientation and foot placement.

### 10.3 Proposed gesture grammar

| Physical intention | Trackpad action | Physical controller action |
|---|---|---|
| Neutral glide | two stable contacts | maintain compliant feet and neutral COM |
| Carve | move both contacts laterally | move desired COM laterally; deck lean follows |
| Weight nose/tail | move centroid fore/aft | shift pelvis and foot normal-load targets |
| Push | hold board foot; sweep other finger backward and return | ground-foot trajectory and push force |
| Crouch | reduce stance length and/or draw both contacts inward/downward by calibrated convention | lower pelvis target, preload legs |
| Ollie | crouch, rear lift/strike/replant motion, front slide | tail strike + unweight + front-foot leveling |
| Nollie | mirrored front-foot strike | nose strike sequence |
| Kick/heelflip | front-foot diagonal edge flick | tangential foot-edge impulse then release |
| Shuv | rear-foot curved/tangential sweep | tail yaw impulse |
| Catch | replant one/both contacts over catch window | foot-deck compliant catch |
| Manual | fore/aft load shift with both contacts | move COM over rear/front truck |
| Revert/pivot | low-speed unload one truck + rotate stance | physical pivot mode |
| Bail/recover | contact loss or impossible state | release constraints, ragdoll/proxy fall |

The exact gestures require user testing. The key invariant is that every gesture drives a physically meaningful target.

### 10.4 Recognition should annotate, not actuate

The recognizer remains useful for:

- UI labels;
- scoring;
- tutorial feedback;
- phase priors;
- choosing an assistance envelope;
- analytics.

It should not be the principal source of board momentum.

A robust split:

```text
continuous trajectory path
    -> physical forces every substep

recognizer
    -> semantic label + confidence + optional bounded assistance policy
```

### 10.5 Input resampling

Preserve every hardware sample. For each physics substep:

- reconstruct a monotone piecewise-cubic or bounded Hermite trajectory;
- clamp impossible velocity/acceleration spikes;
- retain exact lift/replant edges;
- report reconstruction error and sample age;
- avoid averaging away the peak of a fast flick.

EA's 120 Hz input/60 Hz physics precedent shows that raising the outer physics rate is not a substitute for preserving gesture shape.

## 11. Timestep and solver strategy

### 11.1 Recommended rates

Start with:

- input capture: native ~125 Hz or higher when available;
- game/control tick: 60 Hz;
- contact/foot/truck substeps: 2–4 per tick, yielding 120–240 Hz;
- render: display rate with interpolation.

HUSKY uses a 5 ms physics step. That does not prove SlackPad must run at 200 Hz, but it reinforces that contact-rich articulated systems benefit from smaller steps.

### 11.2 Why not simply change `physics.hz` to 240

A raw rate increase will:

- alter every gain expressed per step;
- change friction and damping behavior;
- invalidate replay baselines;
- multiply CPU cost;
- leave the direct heading servo and missing rider untouched.

Use explicit substep ownership:

```ts
for each 60 Hz game step:
  consume/reconstruct input trajectory
  for each physics substep:
    update contact modes
    evaluate rider/foot actuators
    evaluate truck and wheel forces
    integrate Rapier
  emit one game observation + detailed substep telemetry
```

### 11.3 Convergence tests

Run the same scripted physical intent at:

- 60 Hz × 1;
- 60 Hz × 2;
- 60 Hz × 4;
- 120 Hz × 1.

Compare:

- path;
- yaw;
- roll;
- wheel loads;
- pop height;
- board rotations;
- contact impulses;
- energy residual;
- outcome.

Set tolerances before tuning.

## 12. Physical calibration and system identification

A “realistic” simulator without measured reference data becomes a committee of opinions. Build a small empirical dataset.

### 12.1 Hardware kit

Minimal:

- one or more real skateboards;
- phone IMU;
- high-frame-rate phone video;
- printed scale/reference grid;
- rider mass/height;
- optional inexpensive load cells or force plate;
- known ramp/rail geometry.

### 12.2 Tests

#### Board-only

1. **Mass and inertia**
   - weigh deck/trucks/wheels;
   - estimate inertia from geometry or pendulum test.

2. **Free roll**
   - initial speed vs distance/time;
   - fit bearing and rolling resistance.

3. **Truck free-decay roll**
   - estimate stiffness/damping/friction.

4. **Lean-to-steer**
   - measure truck angle vs deck roll.

5. **Carve radius**
   - speed × lean grid;
   - fit geometry/slip correction.

6. **Wheel impact**
   - curb heights and approach speed;
   - validate swept contact.

#### Rider–board

1. **Push**
   - stance, sweep duration, acceleration profile.

2. **Pumping**
   - COM timing vs speed gain on transition.

3. **Ollie**
   - rider COM, tail strike, board pitch, board/rider separation, apex.

4. **Flip**
   - foot path, release, board angular velocity.

5. **Catch**
   - relative board/foot speed and catch impulse.

6. **Landing**
   - wheel loads, leg compression, speed loss, stability.

### 12.3 Fitting

Use a staged objective:

\[
J(\theta)=
w_p J_{\text{path}}
+w_q J_{\text{orientation}}
+w_v J_{\text{velocity}}
+w_c J_{\text{contact}}
+w_e J_{\text{energy}}.
\]

Fit low-dimensional subsystems first:

1. wheel rolling loss;
2. truck stiffness/damping;
3. carve geometry/slip;
4. rider/leg compliance;
5. push;
6. pop/flick/catch.

Methods:

- bounded least squares for simple tests;
- CMA-ES or Bayesian optimization for nonsmooth contact parameters;
- differentiable physics only where gradients are trustworthy;
- domain randomization around fitted uncertainty, not around arbitrary ranges.

### 12.4 Behavioral reference from commercial games

Commercial games can also be measured without copying code or assets:

- capture controlled input/video at known frame rate;
- run straight-line acceleration, carve, pop, flip, catch, and landing experiments;
- estimate response surfaces;
- compare latency, angular velocity, damping, and failure envelopes.

This is black-box behavioral benchmarking. It should be used to match *feel*, not to claim the underlying implementation is identical.

### 13.2 Energy and momentum ledger

For each substep, account for:

\[
\Delta E =
W_{\text{gravity}}
+W_{\text{rider}}
+W_{\text{assist}}
-W_{\text{dissipation}}
+\epsilon_{\text{numerical}}.
\]

During free flight with no assist:

- linear momentum changes only under gravity;
- angular momentum about COM should be approximately conserved;
- unexplained energy should remain within tolerance.

This ledger turns “it popped weird” into a debuggable statement.

### 13.3 Contact topology

Record transitions such as:

```text
rear-foot-board -> rear-foot-air
tail-ground begins
front-foot-board sliding
all-wheel-ground -> airborne
front-foot-board catch
rear-foot-board catch
wheel-ground landing
```

A trick is primarily a contact sequence. Recording only `phase = air` destroys the explanation.

### 13.4 Assist ledger

Every assist event should include:

- source controller;
- requested force/torque;
- applied force/torque;
- impulse;
- power;
- cumulative work;
- saturation reason;
- assist profile.

A user or agent should be able to replay a trick with assistance visualization turned on and see exactly where the game helped.

## 14. Programmatic self-research for Codex or another coding agent

The repository already has the right security boundary. Extend it into a scientific loop.

### 14.1 Scenario runner

Create deterministic headless scenarios:

```text
physics/scenarios/
  static-settle
  free-roll-decay
  truck-roll-decay
  carve-radius-grid
  push-acceleration
  low-speed-pivot
  bank-transition
  curb-impact
  ollie-sequence
  kickflip-sequence
  shuv-sequence
  catch-sequence
  landing-drop
  grind-entry
  grind-exit
```

Each scenario should define:

- initial state;
- ContactFrame/foot-target input trace;
- environment;
- parameter set;
- expected invariants;
- output metrics;
- optional reference trajectory.

### 14.2 Invariant tests

Examples:

- no direct yaw torque in raw glide mode;
- at zero speed and all wheels loaded, ordinary carve input cannot create large yaw;
- regular/goofy mirrored inputs produce mirrored paths;
- wheel spin and translation agree in pure rolling;
- free-flight angular momentum drift stays below threshold;
- controller force and power never exceed configured bounds;
- assistance work is zero in raw mode;
- no NaN/Inf under randomized inputs;
- contact impulses remain finite;
- substep convergence passes;
- deterministic replay hashes match within one build;
- no trick completes from label alone without required physical contact sequence.

### 14.3 Metrics

#### Ride

- acceleration profile;
- rolling decay time;
- carve radius vs speed/lean;
- lateral slip;
- wheel-load distribution;
- steering latency;
- push impulse and cadence;
- pumping energy gain.

#### Tricks

- pop height;
- tail strike time;
- pitch at takeoff/apex;
- board-rider separation;
- flip/shuv angular impulse;
- angular momentum drift;
- catch relative speed;
- landing impulse;
- stability margin;
- assist work.

#### Control

- input-to-force latency;
- input-to-visible-response latency;
- contact-role stability;
- gesture trajectory reconstruction error;
- false phase transitions;
- confidence calibration.

#### Numerical

- energy residual;
- penetration;
- solver residual;
- substep sensitivity;
- CPU time;
- worst-case allocations.

### 14.4 Agent workflow

A coding agent should operate as a laboratory assistant:

1. select one failed metric;
2. generate or reuse a minimal deterministic scenario;
3. inspect trace and ledger;
4. state a falsifiable hypothesis;
5. modify one subsystem or parameter family;
6. rerun scenario matrix;
7. run full regression suite;
8. export before/after traces and plots;
9. open a small PR with the hypothesis and evidence;
10. request human A/B only after objective gates pass.

Do not let an agent simultaneously retune steering, friction, camera, and gesture thresholds. That produces an impressive demo and an uninterpretable system.

### 14.5 Parameter search

Allow agents to run bounded experiments:

```yaml
experiment:
  scenario: carve-radius-grid
  objective:
    - path_rmse
    - yaw_rate_rmse
    - energy_residual
  parameters:
    front_truck.k1: [8, 25]
    rear_truck.k1: [8, 25]
    truck.damping: [0.2, 2.0]
    wheel.cornering_stiffness: [20, 200]
  constraints:
    max_assist_work: 0
    max_penetration_mm: 2
    realtime_factor_min: 1.5
```

Use held-out scenarios to prevent overfitting.

### 14.6 Human playtesting

Physics metrics do not fully determine feel. Use blinded paired comparisons:

- baseline vs candidate;
- same level, camera, and input tutorial;
- random order;
- questions:
  - “Which feels more connected to the ground?”
  - “Which makes your mistake easier to understand?”
  - “Which better matches the gesture you made?”
  - “Which is more like *Skate*?”
  - “Which would you keep playing?”

Fit Bradley–Terry or TrueSkill-style preference scores and retain raw comments.

Require both:

- objective physical improvement;
- human preference improvement.

A suggested promotion gate is at least 65% blinded preference over baseline with no major regression in controllability or performance.

### 14.7 Visual playtesting

Add automated video capture overlays:

- foot targets and actual feet;
- COM projection;
- contact points/normals;
- wheel loads;
- truck steer;
- slip vectors;
- assist force arrows;
- energy residual;
- phase timeline.

A browser agent can then inspect both telemetry and frames. This is much more useful than asking a vision model whether a clip “looks realistic” without physical context.

- swept wheel contacts;
- 2–4 contact substeps.

Keep legacy rider-less control temporarily to isolate the subsystem.

Acceptance:

- free-roll and roll-decay tests fit reference data;
- carve radius is monotonic in speed and lean;
- zero-speed ordinary lean produces negligible yaw;
- 60/120/240 Hz metrics converge;
- curb impacts are stable.

### Phase 2 — Minimal rider and physical push

Implement:

- rider proxy mass/inertia;
- pelvis/COM controller;
- two compliant feet;
- foot-board and foot-ground modes;
- trackpad common-mode/differential intent;
- physical push;
- weight transfer and pumping.

Acceptance:

- static wheel loads sum to rider+board weight;
- push acceleration falls in measured range;
- rider mass changes acceleration/load naturally;
- carve emerges from COM lean;
- raw mode has no wheel motor;
- pumping can add energy only through rider work.

### Phase 3 — Physical pop, flick, shuv, and catch

Implement:

- crouch/unweight sequence;
- tail/nose strike;
- front-foot slide;
- edge flick;
- board separation;
- catch contact;
- raw-mode removal of midair target-angular-velocity PD;
- semantic recognizer retained for UI/assists.

Acceptance:

- no flip without foot-edge impulse;
- no shuv without tail tangential impulse;
- flight angular momentum residual below threshold;
- catch impulse depends on relative motion;
- pop height/pitch fit reference envelopes;
- gesture magnitude continuously affects outcome.

### Phase 4 — Landing, bail, and grind

Implement:

- leg-compliance landing;
- support/stability calculation;
- physical speed loss;
- contact-manifold grind/slide;
- assist-budgeted grind capture;
- improved bail body behavior.

Acceptance:

- no fixed horizontal-speed deletion in raw mode;
- landing loss is explained by contact/leg work;
- rails use correct hanger/deck contacts;
- raw grind has zero magnetic work;
- assisted grind work stays under cap and is visible.

### Phase 5 — Active ragdoll and learned/offline controllers

Optional:

- multi-body rider;
- motion-prior style controller;
- offline MuJoCo/MJX/Isaac training;
- policy distillation to compact deterministic controller;
- board/rider setup variation.

Do this only after the reduced model and measurement suite are stable. A full ragdoll before observability is a cathedral built on fog.

## 17. Acceptance gates for “more like Skate”

A candidate architecture should not be promoted because one kickflip clip looks good.

### Physical gates

- raw-mode yaw controller contributes zero direct chassis torque;
- ordinary zero-speed carve input produces near-zero yaw;
- carve response follows measured speed/lean dependence;
- wheel loads sum to supported weight within tolerance;
- push momentum is transferred through a ground foot;
- unassisted free-flight angular momentum drift < 1–2%;
- unexplained energy residual remains bounded;
- raw-mode assist work is zero or explicitly negligible;
- standard-mode assist work stays below declared budgets;
- contact/substep convergence passes;
- no NaN, runaway impulse, or tunneling across randomized stress tests.

### Control gates

- accepted gesture magnitude maps continuously to physical impulse;
- no 72%-minimum trick speed floor in raw mode;
- role swaps remain below threshold;
- lift/replant timing survives sample jitter;
- regular/goofy mirror tests pass;
- input-to-response latency is measured and bounded.

### Human gates

- majority preference over baseline for ground connection;
- failures are more understandable;
- expert players can intentionally vary pop/flip magnitude;
- novices can use standard assist without feeling random;
- the camera preserves line and landing readability.

## 18. Engine decision

### Recommended now: Rapier + custom reduced-order mechanics

Advantages:

- preserves deterministic replay and browser deployment;
- minimizes unrelated migration risk;
- supports rigid bodies, colliders, queries, and joints;
- lets the team replace the causal model incrementally.

### Recommended offline: MuJoCo/MJX, Nimble, or Dojo

Use for:

- higher-fidelity articulated experiments;
- optimization and identification;
- learned controllers;
- sensitivity analysis.

### Do not select an engine by brand reputation

The original *Skate* evidence shows that architecture and active control mattered. *Session* shows that feel can become coupled to engine details. Neither implies that replacing Rapier automatically creates realism.

The decision rule should be empirical: if the proposed articulated rider-board scenarios cannot meet stability, performance, or contact requirements in Rapier after the reduced model is implemented, run the same scenario suite in candidate engines and compare. Migrate only with evidence.

## 19. What not to do

1. **Do not merely increase physics Hz.** It will make the wrong causal model more expensive.

2. **Do not simply increase `boardMass` to rider+board mass.** It will fix cruising inertia while breaking board separation and trick rotation.

3. **Do not keep absolute heading as the primary steering input.** At most it should become a high-level body-heading preference that must be realized through lean/contact.

4. **Do not tune yaw torque against lateral friction until it “looks right.”** Two compensating nonphysical systems can fit one test and fail everywhere else.

5. **Do not train an opaque RL policy directly in the shipping runtime before building metrics.** It may learn controller exploits and numerical artifacts.

6. **Do not rely on pressure.** The audited raw adapter does not currently emit it.

7. **Do not eliminate all assistance.** Replace hidden assistance with bounded, observable assistance.

8. **Do not clone proprietary code or assets.** Reproduce public behavior and physical principles through clean implementation.

9. **Do not use semantic trick labels as the source of momentum.** Labels should explain and assist physical action, not substitute for it.

10. **Do not let camera changes mask physics regressions.** Test fixed-camera physical metrics and player-facing camera feel separately.

## 21. Final recommendation

SlackPad should become a physics instrument rather than a trick recognizer connected to a board.

The shortest path to *Skate*-like weight is not a full rewrite. It is this causal inversion:

### Current

```text
gesture label / heading
    -> target board motion
    -> servo impulses
    -> wheel solver tries to support the result
```

### Target

```text
trackpad trajectories
    -> feet and rider COM targets
    -> bounded physical forces
    -> truck and wheel contacts
    -> board/rider motion
    -> semantic recognition and scoring
```

Preserve the deterministic spine. Replace the muscles and nerves.

## Appendix A — Quantitative audit notes

### A.1 Yaw inertia and servo authority

\[
I_y = \frac{2.4(0.8^2+0.2^2)}{12}=0.136\,\mathrm{kg\,m^2}
\]

\[
\alpha_{\max} = 28/0.136 = 205.9\,\mathrm{rad/s^2}
\]

\[
\Delta\omega_{\text{one 60 Hz step}}=205.9/60=3.43\,\mathrm{rad/s}
\]

### A.2 Long-axis inertia and flip authority

\[
I_{\text{long}} =
\frac{2.4(0.2^2+0.05^2)}{12}
=0.0085\,\mathrm{kg\,m^2}
\]

\[
\Delta\omega = J/I
\]

- 0.08 N·m·s -> 9.4 rad/s
- 0.14 N·m·s -> 16.5 rad/s
- 0.20 N·m·s -> 23.5 rad/s

### A.3 Pop height

\[
v_0=J/m
\]

- 5.8 N·s / 2.4 kg = 2.42 m/s
- 9.6 N·s / 2.4 kg = 4.0 m/s

\[
h=v_0^2/(2g)
\]

- approximately 0.30–0.82 m for the board-only idealization.

### A.4 Rider-load mismatch

- empty board weight: \(2.4\times9.81=23.5\) N;
- 70 kg rider + board: \(72.4\times9.81\approx710\) N;
- current four-wheel suspension hard cap: \(4\times80=320\) N.

### A.5 Lateral velocity cancellation

At 60 Hz:

\[
g=\mathrm{clamp}(30/60,0,1)=0.5
\]

Across four wheels, absent friction-cap saturation, the solver removes approximately 50% of lateral chassis velocity per step, corresponding to a decay time constant near 24 ms.
